# Chinese Checkers 3D 项目汇报文档

## 1. 项目概述

### 1.1 项目名称

Chinese Checkers 3D，基于 Godot 4.6 开发的 3D 跳棋游戏项目。

### 1.2 项目定位

本项目以经典六角星形跳棋为核心玩法，扩展了 3D 棋盘表现、被动技能棋子、本地人机、在线联机、对局记录/回放、昼夜光照、可调渲染档位和花园场景等内容。项目目标不是只做一个规则演示，而是做成一个具备完整游戏流程、可演示、可扩展、可继续迭代的 3D 棋类游戏原型。

### 1.3 当前完成度概览

- 完成星形跳棋棋盘生成、棋子初始化、棋子移动、连续跳跃、胜负判断。
- 完成基础 3D 棋盘、棋子、格子高亮、可落点提示、被动技能提示。
- 完成本地玩家与 AI 对战流程，并支持 AI 接管。
- 完成 WebSocket 信令 + WebRTC 数据通道的在线对战基础框架。
- 完成对局记录、保存、载入、回放、回退等辅助系统。
- 完成玩家棋子颜色调色盘和互斥选择，避免不同玩家使用同色。
- 完成可切换的昼夜光照、Forward+ 渲染档位、灯光预算控制和场景装饰。
- 完成开始菜单、暂停菜单、灯光设置菜单、材质选择、回放控制等 UI。
- 完成摄像机聚焦、手动缩放/旋转、玩家视角定位和命名视角保存。

## 2. 技术栈与运行环境

### 2.1 引擎与语言

- 游戏引擎：Godot 4.6。
- 脚本语言：GDScript。
- 渲染方式：Forward+。
- 主场景：`res://scenes/Main.tscn`。
- 项目配置：`project.godot`。

### 2.2 主要资源类型

- 3D 场景节点：`Node3D`、`Area3D`、`MeshInstance3D`、`Camera3D`、`Light3D`。
- UI 节点：`CanvasLayer`、`Control`、`PanelContainer`、`Button`、`Label`、`OptionButton`、`Slider`。
- 棋盘与棋子：程序化生成网格与材质，少量场景文件作为基础模板。
- 环境资源：草地贴图、大理石/洞石棋盘材质、Kenney 低多边形自然物件、导入树木与蘑菇资源。
- 音频资源：背景循环音乐与移动、跳跃、技能反馈音效。

### 2.3 项目运行特点

- 固定最大帧率 60 FPS。
- 开启垂直同步。
- 保持 Forward+ 渲染器，用灯光数量预算控制性能成本。
- 关键设置通过 Godot `user://` 保存，例如灯光配置、渲染档位、摄像机视角、对局记录。

## 3. 项目结构

### 3.1 顶层结构

```text
project.godot                     Godot 项目配置
scenes/Main.tscn                  主场景
scenes/Cell.tscn                  棋盘格模板
scenes/Piece.tscn                 棋子模板
scripts/                          游戏核心脚本
scripts/gameplay/MatchRecorder.gd 对局记录模块
assets/                           贴图、模型、音频、UI 资源
docs/                             设计计划与汇报文档
```

### 3.2 主场景节点结构

`Main.tscn` 采用管理器式节点组织方式：

```text
Main
├── BackgroundManager        场景、灯光、渲染档位、环境装饰
├── BoardManager             棋盘、棋子、材质、格子状态
├── ViewFocusMarkerManager   六方视角定位标记
├── MoveValidator            标准跳棋走法校验
├── PassiveSkillRuleEngine   被动技能规则扩展
├── AudioManager             音乐和音效
├── NetworkManager           联机信令和 WebRTC
├── GameManager              游戏主流程与回合控制
├── AIPlayer                 AI 决策
├── Camera3D                 聚焦摄像机
├── DirectionalLight3D       旧主光，实际由 BackgroundManager 管理光照
└── GameUI                   HUD、菜单、设置、回放控制
```

这种结构的优点是职责清晰：规则、棋盘、渲染、UI、网络、AI 分别由独立脚本负责，`GameManager` 只做流程协调。

## 4. 核心架构设计

### 4.1 管理器分层

项目采用“场景管理器 + 信号通信 + 状态快照”的架构。

| 模块 | 文件 | 主要职责 |
| --- | --- | --- |
| GameManager | `scripts/GameManager.gd` | 游戏流程、回合、胜负、AI、联机、回放协调 |
| BoardManager | `scripts/BoardManager.gd` | 棋盘生成、棋子生成、坐标转换、材质与高亮 |
| MoveValidator | `scripts/MoveValidator.gd` | 标准跳棋移动、相邻步、跳跃、连续跳可达范围 |
| PassiveSkillRuleEngine | `scripts/PassiveSkillRuleEngine.gd` | 被动技能扩展规则、行动生成、技能效果预览 |
| AIPlayer | `scripts/AIPlayer.gd` | AI 候选行动搜索与评分 |
| NetworkManager | `scripts/NetworkManager.gd` | WebSocket 信令、WebRTC 连接、消息收发 |
| MatchRecorder | `scripts/gameplay/MatchRecorder.gd` | 对局记录 JSON 的创建、追加、保存、载入 |
| GameUI | `scripts/GameUI.gd` | HUD、菜单、材质、灯光、回放和网络 UI |
| FocusCamera | `scripts/FocusCamera.gd` | 摄像机聚焦、缩放、旋转、视角保存 |
| BackgroundManager | `scripts/BackgroundManager.gd` | 世界环境、昼夜、灯光预算、场景装饰、渲染档位 |
| AudioManager | `scripts/AudioManager.gd` | 背景音乐和动作音效 |

### 4.2 信号驱动

项目大量使用 Godot 信号降低模块耦合。例如：

- `BoardManager` 发出 `cell_clicked`、`piece_clicked`，由 `GameManager` 判断是否可操作。
- `GameUI` 发出 `restart_requested`、`end_turn_requested`、`material_selected`、`lighting_preset_selected` 等信号，`GameManager` 再转发或调用对应模块。
- `NetworkManager` 发出 `online_session_started`、`game_message_received`，`GameManager` 根据主机/客户端角色处理权威状态。
- `BackgroundManager` 发出 `lighting_settings_changed`、`render_cost_profile_changed`，UI 同步显示当前参数。

这样 UI 不直接修改棋盘规则，网络不直接移动棋子，规则系统也不依赖界面表现。

### 4.3 状态快照

项目使用状态快照支撑多个功能：

- 回退一步或回退当前回合内的操作。
- AI 搜索时模拟棋盘状态。
- 对局记录和回放恢复初始局面。
- 联机时进行权威状态同步。

棋子状态通常包含：

- 玩家编号。
- 棋盘坐标。
- 棋子唯一 ID。
- 被动技能 ID。
- 材质/颜色状态。

这种设计避免了“只能移动真实节点”的限制，使规则判断和模拟更容易测试，也方便后续扩展存档、回放、观战等功能。

## 5. 棋盘系统实现

### 5.1 六角坐标体系

项目使用 `Vector2i` 表示轴向六角坐标。六个方向定义在 `MoveValidator.gd` 与 `PassiveSkillRuleEngine.gd` 中：

```text
(1, 0), (1, -1), (0, -1), (-1, 0), (-1, 1), (0, 1)
```

这种坐标体系适合六角格游戏，优点是：

- 相邻格计算简单。
- 跳跃落点可以通过 `direction * 2` 直接计算。
- 连续跳跃可用递归或搜索遍历可达格。
- 星形棋盘的中心区与六个角区都能用坐标规则生成。

### 5.2 星形棋盘生成

`BoardManager.gd` 负责生成中心六边形和六个三角角区。它维护：

- `cells`：所有合法棋盘格。
- `pieces`：当前棋子实例。
- `player_starts`：各玩家起始区域。
- `player_targets`：各玩家目标区域。

棋盘格由 `Cell.gd` 管理，每个格子是可点击的 `Area3D`，包含：

- 六边形网格。
- 碰撞体。
- 基础材质。
- 高亮状态。
- 技能范围状态。
- 分析模式状态。

### 5.3 坐标到世界位置

`BoardManager.coord_to_world()` 将六角坐标转换为 3D 世界位置。棋盘格、棋子、路灯、玩家角落标记都依赖这套转换逻辑，因此棋盘视觉对象和规则坐标保持一致。

这是项目中非常重要的统一点：只要棋盘坐标正确，所有与棋盘绑定的 3D 物体都能稳定定位。

### 5.4 棋盘材质

当前棋盘固定使用大理石/Travertine003 相关材质，避免运行时在多个风格之间摇摆导致画面不统一。棋盘格顶部使用较清晰的石材纹理，侧面使用更粗糙的材质表现厚度。

材质系统同时保留可扩展结构，后续可添加更多棋盘主题，但当前默认风格服务于“明亮花园棋盘”的整体视觉目标。

## 6. 棋子系统实现

### 6.1 棋子节点

每个棋子由 `Piece.gd` 控制，本质是一个 `Area3D`，支持鼠标点击。核心字段包括：

- `player_id`：所属玩家。
- `coord`：棋盘坐标。
- `piece_id`：棋子唯一标识。
- `passive_skill_id`：被动技能类型。

棋子的位置由棋盘坐标转换得到，棋子高度根据半径抬升，避免嵌入棋盘。

### 6.2 棋子造型变化

项目不是所有棋子都使用完全相同模型，而是根据棋子或技能生成不同基础形态，例如：

- 普通圆形棋子。
- 低矮圆顶。
- 较高棋子。
- 棱面宝石形态。
- 冲刺类特殊形态。
- 免疫类特殊形态。

这样可以在不引入大量外部模型的情况下，让棋子阵营和技能产生更高辨识度。

### 6.3 棋子颜色与互斥锁

玩家棋子使用内置基础颜色调色盘，例如红、蓝、黄、绿、橙、紫、青、粉等。`BoardManager` 负责应用玩家材质，`GameUI` 负责显示选择项。

为避免多人局中不同玩家选择相同颜色，项目实现了玩家颜色互斥逻辑：

- 当前玩家可以保留自己的颜色。
- 其他玩家已经选择的颜色会在 UI 中禁用。
- `BoardManager` 也会在应用材质时进行锁检查，防止绕过 UI 导致重复颜色。

### 6.4 棋子技能视觉

棋子会显示技能标签和轻量视觉提示，例如光环、跳跃、免疫等。近期优化方向是降低技能特效密度，避免遮挡棋盘视野，使效果更清新、更适合棋类游戏。

## 7. 规则与回合系统

### 7.1 标准移动规则

`MoveValidator.gd` 实现标准跳棋行动：

- 普通移动：移动到相邻空格。
- 单次跳跃：隔着一个有棋子的格子跳到后方空格。
- 连续跳跃：跳跃后如果仍有可跳目标，可以继续操作同一棋子。

核心判断依据是：

- 目标格必须存在于合法棋盘格集合中。
- 目标格必须为空。
- 跳跃时中间格必须有棋子。

### 7.2 连续跳跃处理

`GameManager.gd` 中维护：

- `selected_piece`：当前选中棋子。
- `legal_targets`：当前可落点。
- `_legal_actions`：当前可执行行动字典。
- `forced_chain_piece`：连续跳跃期间被强制继续操作的棋子。
- `_turn_pending_entries`：当前回合内待提交的行动记录。

当玩家执行跳跃后，系统会检查该棋子是否还能继续跳。如果还能跳：

- 当前回合不立即结束。
- 只允许继续操作同一棋子。
- UI 显示继续跳跃提示。

### 7.3 回合结束

一个回合可能由一步普通移动组成，也可能由多次跳跃组成。回合结束时：

- 提交当前回合内的行动记录。
- 判断当前玩家是否获胜。
- 切换到下一玩家。
- 刷新 UI 与摄像机焦点。
- 如果下一个玩家由 AI 控制，则自动触发 AI 行动。

### 7.4 胜利条件

胜利判断基于目标区域：当某玩家的棋子全部进入对应目标区域，即判定获胜。目标区域由 `BoardManager` 在棋盘生成时建立，和玩家起始区域相对应。

## 8. 被动技能系统

### 8.1 设计目标

被动技能系统用于在经典跳棋基础上增加策略变化，但不破坏基础规则。项目将标准移动和技能扩展分离：

- `MoveValidator` 只负责经典跳棋规则。
- `PassiveSkillRuleEngine` 负责技能对行动集合的影响。

这样既能保持基础规则清晰，也方便以后新增技能。

### 8.2 当前技能类型

| 技能 ID | 功能说明 |
| --- | --- |
| `immobilize_aura` | 定身光环，影响周围六格中的其他棋子 |
| `dash_jump` | 冲刺跳，可在特定条件下跳过连续两个棋子 |
| `freeze_immune` | 免疫定身影响 |
| 空字符串 | 普通棋子，无被动技能 |

### 8.3 行动生成

`PassiveSkillRuleEngine.get_legal_actions()` 返回行动字典，而不是只返回目标坐标。行动字典包含：

- `from`：起点。
- `input_target`：玩家点击的目标。
- `base_landing`：基础落点。
- `final_coord`：最终坐标。
- `move_kind`：移动类型。
- `effects`：额外技能效果。

这种结构比单纯坐标更灵活，可以表示普通移动、标准跳跃、冲刺跳和后续技能效果。

### 8.4 定身与免疫

定身规则通过扫描周围六格实现。如果相邻格存在定身光环棋子，则普通棋子不能主动移动。拥有免疫技能的棋子不受此影响，定身光环棋子自身也不会被自己的效果限制。

### 8.5 技能系统扩展性

未来如果要加入更多技能，例如传送、换位、范围加速等，只需要：

1. 在技能规则引擎中加入新的技能 ID。
2. 扩展行动生成或行动效果。
3. 在棋子视觉中加入对应表现。
4. 在 UI 中补充技能名称与描述。

## 9. AI 系统

### 9.1 AI 决策流程

`AIPlayer.gd` 的核心流程是：

1. 读取当前棋盘状态。
2. 遍历当前玩家所有棋子。
3. 为每个棋子生成合法行动。
4. 对跳跃行动继续递归收集跳跃路径。
5. 对候选路径评分。
6. 选择分数最高的行动，加入少量随机性提高变化。

### 9.2 评分因素

AI 评分主要考虑：

- 向目标区域推进的距离收益。
- 跳跃行动的额外奖励。
- 进入目标区域的奖励。
- 后排棋子的推进优先级。
- 随机扰动，避免每局行为完全相同。

### 9.3 AI 的技术特点

AI 没有直接操作 3D 节点，而是依赖状态快照和规则引擎模拟行动。这使 AI 可以复用玩家规则，同时不会因为视觉节点状态导致逻辑混乱。

当前 AI 属于启发式 AI，适合项目原型阶段。后续可扩展为：

- 更深层的局面搜索。
- Minimax 或 Monte Carlo 策略。
- 根据玩家数量调整目标函数。
- 记录历史局面避免重复移动。

## 10. 网络联机系统

### 10.1 网络架构

`NetworkManager.gd` 实现在线会话基础设施：

- WebSocket 用于房间创建、加入、信令交换。
- WebRTC 用于玩家之间的数据通道。
- STUN 服务器用于协助 NAT 穿透。
- 当数据通道不可用时，保留信令转发作为兜底路径。

### 10.2 房间流程

主机流程：

1. 创建房间。
2. 连接信令服务器。
3. 等待对方加入。
4. 创建 WebRTC offer。
5. 交换 SDP 与 ICE。
6. 数据通道打开后开始游戏。

客户端流程：

1. 输入房间码加入。
2. 接收主机 offer。
3. 创建 answer。
4. 交换 ICE。
5. 数据通道打开后同步游戏消息。

### 10.3 权威逻辑

联机模式中，项目保留权威处理思路：

- 主机负责验证并广播关键移动。
- 客户端发送请求或接收远端结果。
- `GameManager` 根据当前是否在线、是否本地可控、是否主机来决定是否允许执行操作。

这种设计能减少双方状态不一致，也方便后续扩展断线重连、观战和回放同步。

## 11. 对局记录、存档与回放

### 11.1 MatchRecorder 模块

`scripts/gameplay/MatchRecorder.gd` 专门负责对局记录数据，不再把记录格式和文件读写散落在 `GameManager` 内部。

它负责：

- 初始化对局元数据。
- 保存初始棋盘快照。
- 追加行动条目。
- 标记最后一条记录的胜负结果。
- 保存到 `user://match_records/latest_match.json`。
- 从 JSON 载入最新记录。

### 11.2 记录内容

对局记录包含：

- 布局名称。
- 游戏模式。
- 玩家数量。
- 随机种子。
- 技能开关。
- 初始棋盘快照。
- 每一步行动条目。
- 回合结束信息。
- 胜负结果。

### 11.3 回放系统

回放流程是：

1. 载入对局记录。
2. 恢复初始棋盘快照。
3. 按记录顺序逐步应用行动。
4. UI 显示当前步数、总步数和播放状态。
5. 支持播放、暂停、上一步、下一步、跳转。

由于记录存的是逻辑行动而不是视频，因此文件体积小，也可以用于调试规则问题。

### 11.4 回退系统

项目保留回合内或回合级别的状态快照，用于撤销操作。对于跳棋这种连续跳跃可能较长的游戏，回退功能可以显著提升调试和玩家体验。

## 12. UI 与交互系统

### 12.1 UI 组成

`GameUI.gd` 动态构建了多套界面：

- 左上角当前玩家和状态提示。
- 回合结束、测试布局、重新开始等按钮。
- 开始菜单。
- 暂停菜单。
- 在线房间创建/加入面板。
- AI 接管开关。
- 对局记录与回放控制。
- 材质和玩家颜色选择。
- 灯光预设、时间、渲染档位设置。

### 12.2 UI 与逻辑解耦

UI 不直接改动棋盘，而是发出信号，例如：

- 请求重新开始。
- 请求结束回合。
- 请求保存摄像机视角。
- 请求选择材质。
- 请求切换灯光预设。
- 请求切换渲染档位。

`GameManager` 或对应管理器接收后处理，保证业务逻辑集中。

### 12.3 玩家反馈

项目提供多层反馈：

- 选中棋子变色。
- 合法落点高亮。
- 技能影响区域高亮。
- 被定身棋子显示锁定状态。
- 当前技能名称和说明显示在 UI。
- 当前回合步骤数和撤回可用性显示。
- 胜利面板显示获胜玩家。

## 13. 摄像机系统

### 13.1 聚焦摄像机

`FocusCamera.gd` 控制主摄像机。它支持：

- 聚焦当前玩家。
- 聚焦选中棋子。
- 恢复棋盘视角。
- 六方视角切换。
- 鼠标拖拽旋转。
- 滚轮平滑缩放。
- 根据距离调整俯仰角和 FOV。

### 13.2 视角标记

`ViewFocusMarkerManager.gd` 在棋盘六个角落附近生成可点击的玩家方向标记。每个标记颜色与对应玩家棋子颜色同步，方便未来六人局中快速定位各方区域。

### 13.3 命名视角保存

摄像机支持把当前视角保存到 `user://camera_views.cfg`。这对汇报演示很有用，可以提前保存展示棋盘、技能、夜景、联机场景等视角。

## 14. 渲染、灯光与场景系统

### 14.1 Forward+ 渲染

项目明确保持 Forward+ 渲染器。Forward+ 适合本项目的原因是：

- 需要较多局部灯光表现夜间路灯、玩家角落光源和萤火虫。
- 3D 棋盘视角中灯光数量会影响氛围。
- 可以通过灯光预算和渲染档位控制性能。

### 14.2 BackgroundManager 总控

`BackgroundManager.gd` 是视觉系统最复杂的脚本，负责：

- WorldEnvironment。
- 程序化天空。
- 主太阳光。
- 柔和补光。
- 情绪灯和花园灯。
- 内角路灯。
- 玩家角落发光标记。
- 地面材质。
- 草叶 MultiMesh。
- 池塘、小桥、石头、花丛、蘑菇、树木等场景元素。
- 昼夜时间采样。
- 灯光预设。
- 渲染成本档位。

### 14.3 昼夜系统

项目通过时间关键帧定义不同时间段的环境参数，例如：

- 日出。
- 上午。
- 正午。
- 黄昏。
- 夜晚。

切换时间时，系统根据当前小时插值采样天空、太阳、环境光、雾、灯光强度等参数。这样同一场景可以在白天和夜晚表现出不同氛围。

### 14.4 灯光预设

灯光预设用于快速切换整体画面风格，例如封面风格草地、夜间棋盘、明亮花园等。UI 中可以选择预设，也可以手动调整参数后保存。

### 14.5 渲染成本分档

项目实现了 high、medium、low 三档渲染成本控制。不同档位会影响：

- MSAA、FXAA、TAA 等视口设置。
- 渲染比例。
- 草叶数量。
- 环境装饰密度。
- 局部灯光数量。
- 内角路灯实际启用数量。

其中内角路灯预算保持：

- high：6 盏真实 `SpotLight3D`。
- medium：4 盏真实 `SpotLight3D`。
- low：2 盏真实 `SpotLight3D`。

这种方案保证高档画面完整，低档仍能运行，同时不改变棋盘规则。

### 14.6 内角路灯系统

内角路灯根据棋盘内角锚点生成，仍然依赖 `BoardManager.coord_to_world()` 定位。每盏灯包含：

- 灯杆。
- 灯罩。
- 灯泡材质。
- `SpotLight3D` 光源。
- 可选调试标记。

路灯在夜晚用于点亮棋盘边缘和中心区域，但小灯不启用阴影，避免性能成本过高。

### 14.7 玩家角落光源

六个玩家角落的标记颜色与玩家棋子颜色一一对应。夜晚时这些标记也承担一定视觉引导和照明作用，方便玩家快速找到自己阵营。

### 14.8 场景资源生成

环境物体包括：

- 草地地面。
- 小型植物、花、石头。
- 池塘和水面。
- 小桥。
- 树木。
- 单个随机散布的蘑菇。
- 萤火虫。

场景物体生成时需要避开棋盘、玩家区域和已有大型物体，避免遮挡棋盘或出现堆叠。

## 15. 音频系统

`AudioManager.gd` 负责音频播放。当前音频包括：

- 菜单或游戏中的循环背景音乐。
- 棋子落子音效。
- 普通跳跃音效。
- 冲刺跳音效。
- 冰冻/解除类反馈音效。

音频系统使用音效池和冷却控制，避免短时间大量重复音效造成刺耳或性能浪费。

## 16. 典型数据流程

### 16.1 本地玩家移动流程

```text
玩家点击棋子
→ BoardManager 发出 piece_clicked
→ GameManager 检查当前玩家是否可控
→ PassiveSkillRuleEngine 生成合法行动
→ BoardManager 高亮合法落点
→ 玩家点击目标格
→ GameManager 取出对应 action
→ 规则校验并移动棋子
→ 播放音效和刷新技能状态
→ 如果还能继续跳，进入连续跳跃
→ 否则结束回合并切换玩家
```

### 16.2 AI 回合流程

```text
轮到 AI 玩家
→ GameManager 创建当前状态快照
→ AIPlayer 枚举可行动棋子
→ PassiveSkillRuleEngine 生成行动
→ AIPlayer 搜索跳跃路径并评分
→ 返回最佳行动
→ GameManager 应用行动
→ 判断是否继续跳或结束回合
```

### 16.3 在线移动流程

```text
本地玩家点击操作
→ GameManager 判断是否在线模式
→ 本地发送 move 请求或主机直接验证
→ 主机执行权威校验
→ NetworkManager 广播行动消息
→ 双方 GameManager 应用同一行动
→ 回合推进消息同步
```

### 16.4 回放流程

```text
载入 latest_match.json
→ MatchRecorder 解析记录
→ GameManager 恢复初始快照
→ UI 显示回放控制
→ 每一步按记录 action 应用
→ 可暂停、单步、跳转或退出回放
```

## 17. 重点技术方案总结

### 17.1 六角坐标与棋盘统一定位

项目最大基础是六角坐标。规则、棋盘、棋子、灯光、玩家角标都围绕同一坐标系统工作，避免视觉和逻辑脱节。

### 17.2 行动字典替代单目标坐标

普通棋类移动通常只需要“从哪里到哪里”，但本项目加入技能后，仅用目标坐标不够表达完整行为。因此规则系统返回行动字典，携带移动类型和技能效果，为扩展技能打下基础。

### 17.3 状态快照驱动调试能力

状态快照让项目同时获得 AI 模拟、回放、回退、联机同步的基础能力。这是比单纯操作场景节点更稳健的做法。

### 17.4 渲染预算控制

项目不是盲目堆叠灯光和特效，而是通过 high/medium/low 档位控制：

- 实际光源数量。
- 抗锯齿和渲染比例。
- 草和场景细节数量。

这使视觉效果和运行性能之间可以平衡。

### 17.5 视觉系统不影响规则系统

近期大量工作都围绕棋盘材质、花园场景、路灯、蘑菇和玩家角落标记展开，但这些修改被限制在 `BackgroundManager`、材质和表现层中，不影响棋盘规则、AI、网络、存档和 UI 逻辑。这体现了项目架构的边界控制。

## 18. 技术难点与解决方案

| 难点 | 解决方案 | 收益 |
| --- | --- | --- |
| 六角星形棋盘坐标复杂 | 使用轴向坐标和固定六方向向量 | 移动、跳跃、区域判断统一 |
| 连续跳跃状态容易混乱 | 使用 `forced_chain_piece` 和回合待提交记录 | 玩家操作清晰，记录完整 |
| 被动技能会污染基础规则 | 将技能放到独立规则引擎 | 标准规则可维护，技能可扩展 |
| AI 需要试走但不能移动真实节点 | 基于状态快照模拟行动 | 避免视觉节点和逻辑状态冲突 |
| 联机双方可能状态不一致 | 主机权威验证并广播行动 | 降低同步错误 |
| 对局回放需要可重复 | 保存初始快照、随机种子和行动列表 | 回放轻量、可调试 |
| 夜间灯光数量多 | Forward+ + 灯光预算 | 保持氛围同时控制性能 |
| 场景装饰容易遮挡棋盘 | 使用半径、保留区和随机清理位置 | 装饰服务棋盘，不干扰玩法 |
| 多玩家颜色容易重复 | UI 锁定 + BoardManager 二次校验 | 六人局可读性更好 |

## 20. 当前不足与后续优化方向

### 20.1 玩法方向

- 支持完整六人局配置。
- 为每个玩家提供更明确的阵营 UI 和摄像机快捷入口。
- 增加技能开关、技能池配置和更公平的技能分配策略。

- 完善公网信令服务器部署。
- 增加断线重连。
- 增加房间内玩家准备状态。
- 增加联机状态校验和异常恢复。

### 20.3 表现方向

- 继续降低棋子特效遮挡。
- 优化夜晚棋盘照明，让玩家角落和棋盘中心更自然。

- 加入更柔和的环境雾和后处理，但保持可读性。

### 20.4 工程方向

- 把 `GameManager` 中的回合系统进一步拆分为独立 `TurnController`。
- 给规则引擎补充自动化单元测试。
- 将网络消息协议文档化。
- 为场景生成增加可视化调试开关。

## 23. 总结

Chinese Checkers 3D 当前已经从一个基础跳棋原型扩展为一个包含完整游戏流程的 3D 棋类项目。它的核心价值在于：

- 规则层稳定：六角坐标、跳跃、连续跳跃、胜负判断完整。
- 工程层清晰：各管理器职责明确，信号通信降低耦合。
- 扩展层可持续：技能、AI、联机、回放都建立在状态快照和行动模型上。
- 表现层完整：Forward+、昼夜、灯光预算、花园场景和可调视觉配置提升演示效果。

后续只要继续拆分回合系统、完善六人局和联机稳定性，再持续打磨视觉统一度，本项目可以从课程/阶段性汇报原型进一步发展成完整的小型策略棋类游戏。
